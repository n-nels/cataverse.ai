# Instrument Module Refactor Checklist (Phase 5.3)

This checklist captures the refactoring plan for `src/instrument/opus_wrapper.py`.
The goal is clarity and separation of concerns **without changing behavior**.

## Current Status

Substantial refactoring has been completed, but Phase 5.3 remains **pending approval**
for parity validation and production readiness checks. The work completed so far
includes the structural changes described below (state consolidation, queue-based
analysis dispatch, and helper extraction). Remaining work is focused on parity
verification, any necessary follow-up adjustments, and explicit user sign-off
before production disruption.

## Refactor Plan Checklist

1. **Stabilize interfaces + global state inventory**
   - Identify all globals (pipe handle, paths, counters, file lists).
   - Consolidate into a single runtime state container (`@dataclass`).
   - Preserve naming and behavior; only change access patterns.

2. **Separate OPUS pipe commands from workflow logic**
   - Isolate low-level OPUS command helpers into a dedicated client module/class.
   - Keep command strings unchanged; provide a single send/parse interface.

3. **Extract acquisition workflow**
   - Split measurement, saving, and metadata extraction into explicit steps.
   - Separate post-processing (subtract IFG, peak fitting) from acquisition.
   - Preserve side effects and file outputs exactly.
   - Ensure the workflow uses the refactored analysis modules in `src/analysis/`.

4. **Extract `MessageHandler` into explicit command routing**
   - Introduce `handle_message(message, state)` returning a response string.
   - Implement discrete handlers for background, reset, acquire, readme, end.
   - Remove deeply nested try/except blocks while keeping behavior intact.

5. **Consolidate path/config handling**
   - Replace scattered path globals with a paths container.
   - Resolve paths once per run or message (no recompute churn).

6. **Thread/task handling isolation**
   - Centralize async analysis triggers (peak fit / iso exchange).
   - Use a queue so only **one** thread per data type runs asynchronously.
   - Maintain the same ordering/outputs; treat the queue as a parity-safe guard.

7. **Server loop cleanup**
   - Keep ZMQ loop intact but thin: recv → handle → send.
   - Avoid business logic inside the polling loop.

## Phase 5.3.A (Proposed Modularization Structure)

Target module layout for `src/instrument/` to make responsibilities explicit while
preserving current behavior. `opus_wrapper.py` is removed in favor of explicit
modules and a clear `main.py` entry point.

```
src/instrument/
├── client.py
├── paths.py
├── state.py
├── dispatch.py
├── acquisition.py
├── server.py
└── main.py
```

### Responsibility Breakdown (Moves)

1. **client.py (OPUS pipe adapter)**
   - `pipe_command`
   - `load_file`, `unload_file`
   - `save_as`, `save_as_scsm`
   - `spectrum_from_interferogram`
   - `read_parameter`, `read_multiple_parameters`
   - `deconvolute`, `get_experiment_path`, `get_version`, `check_instrument_status`

2. **paths.py (path + config assembly)**
   - `OpusPaths`
   - `build_paths`
   - `define_paths` (TBD; keep as helper if still required)

3. **state.py (runtime state)**
   - `OpusState`
   - `STATE`, `get_state`
   - `ensure_paths`, `ensure_queues`

4. **dispatch.py (analysis queues + dispatch)**
   - `AnalysisQueue`, `AnalysisQueues`
   - `build_analysis_queues`
   - `dispatch_analysis`

5. **acquisition.py (measurement workflow)**
   - `do_background_measurement`
   - `do_sample_measurement`
   - `opus_acquire`
   - `subtract_ifg_files`
   - `main_tpd` (removed; legacy-only)

6. **server.py (message routing + ZMQ loop)**
   - `handle_message`, `handle_readme`, `handle_end_experiment`, `handle_background`
   - ZMQ polling loop helpers

7. **main.py (entry point)**
   - `main` (bootstrap and start server)
   - `run_server`, `main`

## Constraints

- Output files, schemas, and filenames must remain unchanged.
- OPUS command strings and semantics must remain unchanged.
- Orchestration order (acquire → process → analyze) must remain unchanged.
- Do not alter the logic for functions that rely on OPUS pipe commands.
- Under no circumstances should `opus_wrapper.py` be executed in this environment.

## Remaining Considerations (Review/Validation)

- Message schema validation remains minimal (Option A): malformed dicts raise and return error responses.
- Option C (future): add an explicit command field with required keys per command for clearer routing.
- Option D (future): introduce a typed schema (e.g., dataclass/Pydantic) with validation and defaults.
